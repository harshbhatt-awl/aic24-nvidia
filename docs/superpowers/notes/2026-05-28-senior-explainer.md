# Multi-camera people tracking on NVIDIA Warehouse data — walkthrough for senior

## What we are building

We are running a multi-camera people-tracking pipeline on the NVIDIA PhysicalAI-SmartSpaces Warehouse dataset, which is the same data NVIDIA released for the 2024 AI City Challenge Track 1. The warehouse scene we work with has seven fixed cameras pointed at a shared floor area, recording the same people from different angles for thirty seconds. Our goal is to produce, for every person who walks through that warehouse, a single consistent identity that follows them across every camera that sees them, plus their location on the warehouse floor in real-world metres. The reference solution we are adapting is the YACHIYO RIIPS pipeline from that NVIDIA challenge — we use it as-is wherever possible and only swap in modern model components where the original ones do not build on current hardware.

## How the pipeline works end to end

The pipeline runs in eight stages, one after the other. The first stage takes the raw NVIDIA dataset and writes it out in a directory layout the original YACHIYO code expects, including a per-camera calibration file. The second stage extracts individual frames from each camera's MP4. The third stage runs a person detector — currently YOLO11-x — and produces a bounding box around every person in every frame. The fourth stage runs the ReID model, which is SOLIDER Swin-Small, and converts each person's bounding box into a 768-dimensional appearance embedding that should be similar for the same person across different camera views. The fifth stage runs a pose estimator, RTMPose-large, which gives us the seventeen body keypoints — head, shoulders, hips, knees, ankles, and so on — so that downstream stages can use the ankle position rather than the centre of the bounding box when computing where on the floor the person is standing. The sixth stage is single-camera tracking: for each camera independently, it stitches per-frame detections into tracks, so we know that the bounding box at frame 100 and the bounding box at frame 101 belong to the same person within that one camera. The seventh stage is multi-camera tracking: it takes the per-camera tracks and matches them across cameras, so that the same person walking from camera A's view into camera B's view gets one global identity rather than two separate ones. The eighth stage compares our predicted global tracks against the ground-truth file using TrackEval and computes the standard tracking metrics — HOTA, MOTA, DetA, AssA, IDF1 — both in image space and, separately, in real-world 3D coordinates on the warehouse floor.

## Calibration and homography, in plain language

To map a pixel a person occupies in a camera image to where that person is actually standing on the warehouse floor in metres, we need to know two things about each camera. The first is the camera's intrinsics — its focal length, its image centre, and any lens distortion — which together describe how the camera's lens turns a point in three-dimensional space into a pixel on its image sensor. The second is the camera's extrinsics — its position in the warehouse and which direction it is pointing, expressed as a rotation and a translation relative to some shared warehouse origin. Put together, these are called the camera's calibration, and the NVIDIA dataset supplies them as a projection matrix (a 3-by-4 matrix that combines the intrinsics, the rotation, and the translation into a single linear operation that takes a 3D point in warehouse coordinates and projects it to a pixel).

A homography is a simpler shortcut that works only when everything you care about lies on a single flat plane in the world. In our case that plane is the warehouse floor. A homography is a 3-by-3 matrix that takes a pixel in the camera image and returns where on the floor (X and Y in metres, with Z assumed to be zero) that pixel corresponds to. It works because, mathematically, projecting a flat plane through a camera lens onto a flat image sensor is a well-behaved linear operation that can be inverted exactly, so long as you trust that the world point really was on that floor plane. The homography is cheap to compute and cheap to apply — one matrix multiplication and a divide. The full calibration handles points at any height; the homography only handles points on the floor.

This matters because of how we project people into world coordinates. If we used the centre of the bounding box, we would be guessing at the height of the person's torso above the floor, which means our world coordinate would be off by however far the camera's line of sight to the torso intersects the floor — sometimes by metres. So instead we use the ankle keypoint coming out of the pose stage. The ankles are (almost) always touching the floor, which means the flat-plane assumption baked into the homography actually holds, and the world coordinate we get back is the actual spot the person is standing on. We tried using the average of the two ankles to be robust to occlusions, but it turns out averaging makes things worse — when one foot is mid-stride and airborne, the average ankle is not on the floor anymore, and the homography assumption breaks down. We now use whichever ankle is lower in the image, because that one is almost always the planted foot.

## Parameter tuning so far

The YACHIYO tracker has a handful of thresholds that control its behaviour, and the right values depend heavily on the dataset. The most important ones are: epsilon_scpt, which controls how strict the single-camera tracker is when deciding two detections in nearby frames belong to the same person (lower is stricter, fewer false merges but more broken tracks); epsilon_mcpt, the same idea but for matching tracks across different cameras; sim_th, the minimum cosine similarity between two tracks' ReID embeddings before the multi-camera tracker is willing to link them as the same person; distance_th, the maximum world-coordinate distance between two tracks before linking is allowed (a sanity check that says "the same person cannot be in two places at once"); short_track_th, which drops any track shorter than this many frames as probably-noise before multi-camera linking even sees it; and keypoint_condition_th, the minimum confidence we require on the ankle keypoint before we trust the world projection that uses it.

To tune these without re-running the slow detector, ReID, and pose stages each time, we built an experiment harness. We lock a single reference run on disk as "the baseline" and define each experiment as a small set of overrides on top of the baseline's config. When we run a variant, the harness symlinks the baseline's outputs for every stage before the one we are tuning, so only the affected downstream stages — usually just single-camera tracking, multi-camera tracking, and evaluation — actually re-execute. A full pipeline run is about forty-five minutes; a tracker-parameter variant is about ninety seconds. That lets us sweep dozens of parameter values per evening.

The tuning loop has gone roughly like this. We started from a v2 baseline that had reasonable defaults inherited from the original NVIDIA YACHIYO config. We ran a single-parameter sweep on epsilon_mcpt and found the best value lifted world HOTA from about 0.48 to about 0.51, so we promoted that as v3. We then swept ankle_avg versus ankle_lower for the floor projection and confirmed the ankle_lower choice is reliably better — by 0.066 HOTA — and we wrote that finding into memory so future agents do not repeat the experiment. We then retuned epsilon_scpt from 0.30 down to 0.20 and got another small lift, which we promoted as v3.1. The current numbers are image HOTA 0.780 and world HOTA 0.5282. We have hit diminishing returns on parameter tuning alone — the remaining gap to higher numbers is now coming from two specific bottlenecks in the model components rather than from tracker thresholds.

## Where we are stuck — back-camera linking

The remaining failure mode is concentrated on three cameras in the warehouse, which we informally call the back cameras (camera 0394, 0395, and 0396). On these cameras, two problems compound. First, YOLO is missing a lot of detections — on camera 0395 it finds roughly sixty-two percent of the people it should, and on camera 0396 only thirty-five percent. Those people exist on those cameras' frames but the detector simply does not see them. Second, even for the people it does detect on those cameras, the ReID embeddings produced for the resulting crops are not similar enough to embeddings of the same people on the front cameras for the multi-camera tracker to link them. Concretely, across the long tracks (a hundred and twenty frames or more) on those three cameras, only zero, one, and one tracks respectively get linked into global identities spanning multiple cameras. The other tracks just stay as orphans, which means the same person walking through the warehouse gets two or three separate identities instead of one. We have two workstreams planned, in order of risk: a detector-recall sweep that varies YOLO's confidence threshold to try to recover the missing detections, and a SOLIDER ReID fine-tune that adapts the embedding model to warehouse camera geometry.

## ReID fine-tune workstream — what it is and why

The SOLIDER ReID model we use was pretrained on a public dataset called MSMT17, which is mostly street-level surveillance footage — cameras at roughly head height, looking out at people walking past. Our warehouse back cameras are mounted higher and look down at oblique angles, so the model has never really seen people from those viewpoints during its training. The embeddings it produces for those views are technically discriminative (different people get different embeddings) but only marginally so — the average cosine similarity between embeddings of the same person on a back camera and a front camera is too low to clear the multi-camera linker's threshold. Pushing that average similarity up is exactly what fine-tuning is designed to do.

The plan is to do a partial fine-tune rather than a full one. The intuition is that the early layers of the backbone — the ones that detect edges, textures, body shapes — are angle-invariant and already work fine. The problem lives in the last stage of the backbone, where the model composes those mid-level features into the high-level appearance representation that the ReID cosine compares. So we freeze the first three stages of the Swin backbone and only train the last stage plus the small normalization layer that produces the final embedding. The training uses SOLIDER's own loss recipe — a classification loss to teach identity, a triplet loss to push embeddings of the same person closer and different people further apart, and a centre loss to keep each identity's embeddings clustered — so we are not introducing any new training tricks the original model was not designed for. We hold out the one scene we evaluate on (scene_044) so we are not training and testing on the same data, and we train on the other warehouse scenes from the NVIDIA release. The whole training run is one to three hours on a free Colab T4 GPU, or thirty to sixty minutes on an A100. The output is a single new checkpoint file, drop-in compatible with the existing pipeline — we change one config knob in the repo to point at the new weights, rebuild only the ReID and downstream stages on the locked baseline, and compare the world HOTA against v3.1. The target is to lift back-cam global ID counts from 0/1/1 to at least 5/5/5 and gain at least 0.01 world HOTA.

## Why this is workstream number six and not number one

We deliberately held the ReID fine-tune for last because it is the most expensive thing on the list and has the most ways to go wrong. Catastrophic forgetting is real — if we train too aggressively the model gets very good at our warehouse and very bad at everything else, which would briefly look like a win on our test scene and then collapse the moment we ran on different data. We have two cheaper workstreams to try first: the detector-recall sweep, which only changes YOLO's confidence threshold and costs roughly four to five hours of pipeline runtime to evaluate five variants; and a per-camera tracker threshold tuning, which costs minutes. If either of those lifts the back-cam linking enough on its own, we may not need to fine-tune at all. The fine-tune workstream is fully designed and saved as a spec document so it is ready to execute whenever the cheaper options run out.
