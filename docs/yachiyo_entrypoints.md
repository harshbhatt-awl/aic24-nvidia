## Data directory conventions

All stages read from and write to a set of directories that must exist **at the
same root** as the upstream repo (or a path passed as `root_path` on the CLI).
The conventions used throughout the codebase:

| Logical name   | Relative path (from `root_path`)         | Direction |
|----------------|------------------------------------------|-----------|
| Original       | `Original/scene_NNN/camera_NNNN/`        | read      |
| Frames         | `Original/scene_NNN/camera_NNNN/Frame/`  | written by extract_frame |
| Detection TXT  | `Detection/scene_NNN/camera_NNNN.txt`    | written by detection |
| Detection JSON | `Detection/scene_NNN/camera_NNNN.json`   | written by detection |
| EmbedFeature   | `EmbedFeature/scene_NNN/camera_NNNN/`    | written by embedding |
| Pose JSON      | `Pose/scene_NNN/camera_NNNN/camera_NNNN_out_keypoint.json` | written by pose |
| Tracking JSON  | `Tracking/scene_NNN/`                    | written by tracking (SCT + MCT) |
| Submission     | `Submission/scene_NNN/track1.txt`        | written by generate_submission.py |

`root_path` is typically the parent directory of the upstream repo itself (i.e.,
`external/` in our layout), **not** the repo root. The scripts set CWD to the
repo root but then pass `../` as `root_path` to the Python scripts.

## extract_frame.sh

**Script body:**
```bash
conda activate botsort_env

for SCENE in $*
do
    F_SCENE=$(printf "%03d" "$SCENE")
    python3 tools/extract_frame.py -s scene_$F_SCENE ./
done
```

- **Python entry:** `python3 tools/extract_frame.py` (relative to upstream repo root)
- **Positional arg:** `root_path` — passed as `./` (the upstream repo root itself)
- **Flag:** `-s / --scene scene_NNN`
- **Inputs read from:** `<root_path>/Original/scene_NNN/<camera_NNNN>/video.mp4`
  — walks every subdirectory of the scene dir, skipping dirs with `"map"` in the name
- **Outputs written to:** `<root_path>/Original/scene_NNN/<camera_NNNN>/Frame/<frame_num>.jpg`
  — zero-padded 6-digit frame numbers, JPEG format
- **Env vars referenced:** None (conda env name is hardcoded as `botsort_env`)
- **Working directory expected:** upstream repo root
  (`external/AIC24_Track1_YACHIYO_RIIPS/`)
- **Notes:**
  - Uses `multiprocessing.Pool` (one process per camera). May exhaust RAM on
    large scenes.
  - Creates `Frame/` subdirectories automatically via `os.makedirs`.
  - `root_path = ./` means the script reads/writes relative to the repo root,
    NOT to the parent data directory. This differs from all other stages.
    **Our orchestrator must pass the repo root as `root_path` for this stage,
    but the data root (parent of `Original/`) for all other stages.**

## detection.sh

**Script body:**
```bash
cp ./detector/aic24_get_detection.py ./BoT-SORT/tools/
cd ./BoT-SORT
conda activate botsort_env

for SCENE in $*
do
    F_SCENE=$(printf "%03d" "$SCENE")
    python3 tools/aic24_get_detection.py -s scene_$F_SCENE ../
done
```

- **Python entry:** `python3 tools/aic24_get_detection.py`
  (CWD = `BoT-SORT/`, which is a sibling of the upstream repo root)
- **Positional arg:** `root_path` — passed as `../` (one level up from `BoT-SORT/`,
  i.e., the same directory that contains the upstream repo and `BoT-SORT/`)
- **Flag:** `-s / --scene scene_NNN`
- **Inputs read from:**
  - `<root_path>/Original/scene_NNN/<camera_NNNN>/Frame/*.jpg` (alphabetically sorted)
  - YOLOX model checkpoint: `bytetrack_x_mot17.pth.tar` (default `--ckpt`, relative
    to CWD = `BoT-SORT/`)
  - YOLOX experiment file: `yolox/exps/example/mot/yolox_x_mix_det.py` (default
    `--exp_file`, relative to CWD = `BoT-SORT/`)
- **Outputs written to:**
  - `<root_path>/Detection/scene_NNN/<camera_NNNN>.txt` — MOT-format CSV:
    `camera_name,frame_id,class,x1,y1,x2,y2,score`
  - `<root_path>/Detection/scene_NNN/<camera_NNNN>.json` — per-detection JSON with
    fields: `Frame`, `ImgPath`, `NpyPath` (initially empty), `Coordinate`, `ClusterID`,
    `OfflineID`
  - Creates `<root_path>/Detection/scene_NNN/` if it doesn't exist
- **CLI args (defaults):**
  - `--exp_file yolox/exps/example/mot/yolox_x_mix_det.py`
  - `--ckpt bytetrack_x_mot17.pth.tar`
  - `--device gpu`
  - `--fps 30`
  - Scale hardcoded: `scale = min(800/1080, 1440/1920)` applied to all bounding boxes
- **Env vars referenced:** None
- **Working directory expected:** `BoT-SORT/` (sibling of upstream repo root)
- **Notes:**
  - Output bounding boxes are clipped to `[0,1920] x [0,1080]` — hardcoded
    resolution. This will produce wrong detections on non-1920x1080 input.
  - The script `cp`s `detector/aic24_get_detection.py` into `BoT-SORT/tools/`
    before running. Our orchestrator must either replicate this copy step or
    point `sys.path` so imports resolve.
  - Output filenames are `camera_NNNN.txt` / `.json` (no `det/` subdirectory).
    **This CONTRADICTS the plan's assumption of `cam_dir/det/det.txt`.**
    Actual path: `Detection/scene_NNN/camera_NNNN.txt` at the root_path level,
    not inside the camera directory.

## embedding.sh

**Script body:**
```bash
cp ./embedder/aic24_extract.py ./deep-person-reid/torchreid/
cd ./deep-person-reid
conda activate torchreid

for SCENE in $*
do
    F_SCENE=$(printf "%03d" "$SCENE")
    python3 torchreid/aic24_extract.py -s scene_$F_SCENE ../
done
```

- **Python entry:** `python3 torchreid/aic24_extract.py`
  (CWD = `deep-person-reid/`, which is a sibling of the upstream repo root)
- **Positional arg:** `root_path` — passed as `../`
- **Flag:** `-s / --scene scene_NNN`
- **Inputs read from:**
  - `<root_path>/Detection/scene_NNN/` — reads every `.txt` file; for each
    finds a matching `.json` file with the same stem
  - `<root_path>/Original/scene_NNN/<camera_NNNN>/Frame/<frame>.jpg` — crops
    per-detection bounding boxes
  - Re-ID model: `<root_path>/deep-person-reid/checkpoints/osnet_ms_m_c.pth.tar`
    (hardcoded path; model auto-downloaded if absent via torchreid)
- **Model:** `osnet_x1_0` (OSNet × 1.0), checkpoint `osnet_ms_m_c.pth.tar`
- **Outputs written to:**
  - One `.npy` file per detection:
    `<root_path>/EmbedFeature/scene_NNN/<camera_NNNN>/feature_<frame>_<uid>_<x1>_<x2>_<y1>_<y2>_<conf>.npy`
  - The `.json` file in `Detection/scene_NNN/` is updated in-place with the
    `NpyPath` field filled for each detection entry
  - Creates `EmbedFeature/scene_NNN/<camera_NNNN>/` if it doesn't exist
- **Env vars referenced:** None (uses `sys.path.append(data_root+'/deep-person-reid')`)
- **Working directory expected:** `deep-person-reid/` (sibling of upstream repo root)
- **Notes:**
  - Embedding filenames encode the full bounding-box coordinates and frame
    number. There is NO single "embeddings file" per camera — there are O(N_det)
    individual `.npy` files.
  - **This CONTRADICTS the plan's assumption of `cam_dir/embeddings/*.np*`.**
    Actual path: `EmbedFeature/scene_NNN/camera_NNNN/*.npy` at the root_path
    level, not inside the `Original/` camera directory.
  - Feature vector dimensionality: 512 (OSNet × 1.0 default).
  - The script iterates over `.txt` files in `Detection/scene_NNN/`, so it
    processes whatever cameras detection produced. Missing cameras are silently
    skipped.

## pose.sh

**Script body:**
```bash
cp ./poser/load_tracking_result.py ./mmpose/demo/
cp ./poser/top_down_video_demo_with_track_file.py ./mmpose/demo/
cd ./mmpose
conda activate openmmlab

for SCENE in $*
do
    F_SCENE=$(printf "%03d" "$SCENE")
    find "../Detection/scene_$F_SCENE" -maxdepth 1 -type f -name "*.txt" | while read -r file;
    do
        CAMERA=$(basename "$file")
        number=$(echo "$CAMERA" | sed 's/camera_\([0-9]\+\).txt/\1/')
        python3 demo/top_down_video_demo_with_track_file.py \
            ../Detection/scene_${F_SCENE}/${CAMERA} \
            ./configs/body/2d_kpt_sview_rgb_img/topdown_heatmap/coco/hrnet_w48_coco_256x192.py \
            https://download.openmmlab.com/mmpose/top_down/hrnet/hrnet_w48_coco_256x192-b9e0b3ab_20200708.pth \
            --video-path ../Original/scene_${F_SCENE}/camera_${number}/video.mp4 \
            --out-file ../Pose/scene_${F_SCENE}/camera_${number}/camera_${number}_out_keypoint.json
    done
done
```

- **Python entry:** `python3 demo/top_down_video_demo_with_track_file.py`
  (CWD = `mmpose/`, which is a sibling of the upstream repo root)
- **Positional args (positional in mmpose demo style):**
  1. `../Detection/scene_NNN/camera_NNNN.txt` — detection file (MOT format, used
     as track file)
  2. `./configs/body/2d_kpt_sview_rgb_img/topdown_heatmap/coco/hrnet_w48_coco_256x192.py`
     — HRNet-W48 config (relative to mmpose CWD)
  3. `https://download.openmmlab.com/mmpose/top_down/hrnet/hrnet_w48_coco_256x192-b9e0b3ab_20200708.pth`
     — checkpoint URL (auto-downloaded by mmcv on first run)
- **Flag:**
  - `--video-path ../Original/scene_NNN/camera_NNNN/video.mp4`
  - `--out-file ../Pose/scene_NNN/camera_NNNN/camera_NNNN_out_keypoint.json`
- **Inputs read from:**
  - `../Detection/scene_NNN/` — uses the same `.txt` files as embedding stage
  - `../Original/scene_NNN/camera_NNNN/video.mp4` — original video (NOT frames)
  - HRNet-W48 COCO checkpoint (downloaded from openmmlab CDN on first call)
- **Outputs written to:**
  - `../Pose/scene_NNN/camera_NNNN/camera_NNNN_out_keypoint.json`
  - Path relative to mmpose CWD = two levels up from mmpose: the same directory
    that contains the upstream repo, BoT-SORT, deep-person-reid, and mmpose
- **Env vars referenced:** None
- **Working directory expected:** `mmpose/` (sibling of upstream repo root)
- **Pose backbone:** **HRNet-W48** (COCO, 256×192, top-down heatmap)
  - Requires mmpose 0.x series (explicitly documented in README; mmpose 1.x has
    a different API)
  - Keypoint schema: **COCO 17 keypoints** (nose, eyes, ears, shoulders, elbows,
    wrists, hips, knees, ankles)
  - VRAM: HRNet-W48 is ~75M parameters; ~3–4 GB VRAM per instance at FP32
- **Notes:**
  - Each camera is processed sequentially (one `python3` invocation per camera).
    No parallelism in this stage.
  - **This CONTRADICTS the plan's assumption of `cam_dir/pose/*.json`.**
    Actual path: `Pose/scene_NNN/camera_NNNN/camera_NNNN_out_keypoint.json` at the
    root_path parent level, not inside the `Original/` camera directory.
  - Output JSON filename includes the camera number extracted by sed from the
    `.txt` detection filename. The sed regex is `camera_([0-9]+).txt`.

## tracking.sh

**Script body:**
```bash
SCENE=$*

for SCENE in $*
do
    echo Procssing scene-$SCENE
    python tracking/infer.py -s $SCENE
done
```

- **Python entry:** `python tracking/infer.py` (relative to upstream repo root)
- **Flag:** `-s / --scene <int>` — scene number as integer (NOT zero-padded)
- **Optional flags (not used by shell script, but available in `infer.py`):**
  - `-o / --output <dir>` — output dir, default `Tracking`
  - `-all / --exec_all` — run both SCT and MCT (default if neither `-scpt` nor
    `-mcpt` is specified)
  - `-scpt / --exec_scpt` — run only SCT
  - `-mcpt / --exec_mcpt` — run only MCT
- **Inputs read from:**
  - `EmbedFeature/scene_NNN/<camera_NNNN>/` — per-detection `.npy` files
    (loaded in `run.run_scpt` via `utils.DetectedObjects.load_from_directory`)
  - `tracking/config/scene_2_camera_id_file.json` — maps scene name to list of
    camera integer IDs (must be present and must include the target scene)
  - `tracking/config/parameters_per_scene.py` — optional per-scene override of
    tracking hyperparameters
  - The tracking code also reads `NpyPath` from the updated Detection JSON files
    (via `scpt.create_similarity_matrix_scpt`)
- **Outputs written to:**
  - SCT (per camera): `Tracking/scene_NNN/camera<NNN>_tracking_results.json`
    — note camera ID is zero-padded to 3 digits (e.g., `camera001`), no
    underscore between "camera" and the number
  - SCT corrected: `Tracking/scene_NNN/fixed_camera<NNN>_tracking_results.json`
  - MCT: `Tracking/scene_NNN/whole_tracking_results.json`
  - MCT corrected: `Tracking/scene_NNN/fixed_whole_tracking_results.json`
  - MCT intermediate: `Tracking/scene_NNN/representative_nodes_scene<N>.json`
- **Env vars referenced:** None
- **Working directory expected:** upstream repo root
  (`external/AIC24_Track1_YACHIYO_RIIPS/`)
- **Splits SCT and MCT? YES**
  - `infer.py` exposes `-scpt` and `-mcpt` flags. When neither is given (as in
    `tracking.sh`), both phases run sequentially in one invocation.
  - Our orchestrator can run SCT-only with:
    `python tracking/infer.py -s <scene_int> -scpt`
  - Our orchestrator can run MCT-only with:
    `python tracking/infer.py -s <scene_int> -mcpt`
  - There is NO env-var switch; control is purely via CLI flags.
- **Embedding path hardcoded:** `embed_path = "EmbedFeature"` in `infer.py`
  line 160 — the embedding root is NOT configurable via `-o`/`--output`. The
  `-o` flag controls only the tracking output directory (default: `Tracking`).
  EmbedFeature root is always `EmbedFeature/` relative to CWD.

## Path overrides our orchestrator will need

Based on findings above, the orchestrator must:

1. **Set CWD to upstream repo root** (`external/AIC24_Track1_YACHIYO_RIIPS/`)
   for `extract_frame.sh` (passes `./` as root_path) and `tracking.sh`. This is
   the only CWD that works without modifying the scripts.

2. **Materialize sibling directories.** Detection, embedding, and pose stages
   each `cd` into a sibling directory (`BoT-SORT/`, `deep-person-reid/`,
   `mmpose/`). Our orchestrator must ensure these are cloned and installed
   adjacent to the upstream repo. Concretely:
   ```
   external/
   ├── AIC24_Track1_YACHIYO_RIIPS/   ← upstream repo root
   ├── BoT-SORT/                     ← required for detection
   ├── deep-person-reid/             ← required for embedding
   └── mmpose/                       ← required for pose
   ```

3. **Copy custom scripts before each stage.** The shell scripts use `cp` to
   inject custom Python files into sibling repos before running. Our orchestrator
   must replicate these copies if calling `infer.py` directly without the shell
   script:
   - detection: `cp detector/aic24_get_detection.py BoT-SORT/tools/`
   - embedding: `cp embedder/aic24_extract.py deep-person-reid/torchreid/`
   - pose: `cp poser/load_tracking_result.py mmpose/demo/` and
     `cp poser/top_down_video_demo_with_track_file.py mmpose/demo/`

4. **Pass `root_path` consistently.** The extract_frame stage uses `./` (repo
   root) as `root_path`, but all other stages use `../` (the parent of the sibling
   dirs, which is `external/`). For our orchestrator to use a unified data root
   it must:
   - For extract_frame: either run with CWD = repo root and pass `./`, OR
     rearrange so `Original/` is symlinked into the repo root.
   - For all other stages: pass the directory one level above all sibling repos
     (i.e., the `external/` directory) as `root_path`.
   - **Recommended:** symlink `external/Original`, `external/Detection`,
     `external/EmbedFeature`, `external/Pose`, `external/Tracking` into
     the upstream repo root so that both `./` and `../` resolve to the same
     data tree.

5. **Override `scene_2_camera_id_file.json` for new scenes.** The tracking
   stage calls `get_camera_ids(scene_id, json_f="tracking/config/scene_2_camera_id_file.json")`
   with no override mechanism. Our orchestrator must either:
   - Patch this JSON file with the NVIDIA scene's camera IDs before running, or
   - Pass a wrapper that injects the correct camera list (requires script modification).

6. **Override `EmbedFeature` path.** In `infer.py` line 160:
   `embed_path = "EmbedFeature"` is hardcoded. The orchestrator must ensure
   CWD contains or symlinks `EmbedFeature/` before calling `tracking/infer.py`.

7. **Harvest outputs** after each stage from fixed locations relative to the
   data root (`external/`), then copy/move them into `outputs/<run_id>/<stage>/`.

## Caveats

### Output path shape diverges from plan assumptions

The plan (Tasks 12-18, Batch 5) contains several incorrect assumptions about
where upstream writes its outputs:

| Stage     | Plan assumed                              | Actual upstream path                                        |
|-----------|-------------------------------------------|-------------------------------------------------------------|
| Detection | `cam_dir / "det" / "det.txt"`             | `Detection/scene_NNN/camera_NNNN.txt` (flat, at data root)  |
| Detection | (no JSON mentioned)                       | `Detection/scene_NNN/camera_NNNN.json` also written         |
| Embedding | `cam_dir / "embeddings" / "*.np*"`        | `EmbedFeature/scene_NNN/camera_NNNN/feature_*.npy` (O(N) files, at data root) |
| Pose      | `cam_dir / "pose" / "*.json"`             | `Pose/scene_NNN/camera_NNNN/camera_NNNN_out_keypoint.json` (at data root) |
| SCT       | `cam_dir / "sct" / "tracks.txt"`          | `Tracking/scene_NNN/fixed_camera<NNN>_tracking_results.json` (JSON, at data root) |
| MCT       | `<scene> / "mct" / "global_tracks.txt"`   | `Tracking/scene_NNN/fixed_whole_tracking_results.json` (JSON, at data root) |

**None of the output paths live inside the `Original/<scene>/<camera>/`
subtree.** They all live at the data-root level in parallel directories:
`Detection/`, `EmbedFeature/`, `Pose/`, `Tracking/`.

**SCT output is JSON, not `tracks.txt`.** The format is a nested dict keyed by
serial number, with `OfflineID` and `Coordinate` fields — not MOT CSV.

**MCT output is JSON, not `global_tracks.txt`.** The final file is
`fixed_whole_tracking_results.json` with camera-keyed nested dicts containing
`GlobalOfflineID`, `Frame`, `Coordinate`, and `WorldCoordinate` fields. The
submission converter (`tools/generate_submission.py`) reads this file and writes
a space-delimited `track1.txt`.

### SCT/MCT phase splitting

`tracking.sh` does NOT split SCT and MCT. Both phases run when `tracking.sh` is
invoked without extra flags. However, `infer.py` supports `-scpt` and `-mcpt`
flags, so our orchestrator CAN split them cleanly by calling `infer.py` directly
with the appropriate flag.

### Conda environments

The scripts hardcode conda environment activation (`conda activate botsort_env`,
`conda activate torchreid`, `conda activate openmmlab`). When running inside a
single container or virtual environment these lines will fail or be no-ops. Our
orchestrator should invoke Python directly (skipping the shell scripts), using
whichever environment has the required packages installed.

### Pose backbone identity

`scripts/pose.sh` uses **HRNet-W48** trained on **COCO** (256×192 input).
- Config: `hrnet_w48_coco_256x192.py` (mmpose 0.x path)
- Checkpoint: `hrnet_w48_coco_256x192-b9e0b3ab_20200708.pth` (~270 MB, auto-downloaded)
- Keypoint schema: COCO 17 keypoints
- mmpose version required: **0.x series** (README explicitly warns that 1.x is incompatible)
- VRAM: ~3–4 GB at FP32 for inference

### Video resolution assumption in detection

`aic24_get_detection.py` hardcodes `scale = min(800/1080, 1440/1920)` and clips
bounding boxes to `[0,1920] x [0,1080]`. The NVIDIA MTMC_Tracking_2024 Warehouse
footage must be 1920×1080 for detection to be geometrically correct. If the
actual footage is a different resolution, detections will be systematically
mis-scaled.

### Pose reads original video, not frames

`pose.sh` passes `--video-path ../Original/scene_NNN/camera_NNNN/video.mp4` — it
reads the video file directly, not the extracted frames. This means the video
file must remain in place even after frame extraction.

### scene_2_camera_id_file.json must be pre-populated

`tracking/infer.py` reads `tracking/config/scene_2_camera_id_file.json` to
resolve which camera IDs belong to a scene. This file ships with AIC24 scene
mappings. For NVIDIA MTMC scenes (different scene IDs), the orchestrator must
prepend or inject an entry for the new scene before calling `infer.py`.
